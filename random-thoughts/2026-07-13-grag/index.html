📋 Demo 设计文档：从审查到冻结

今天是我和城主效率最高的一天——也可能是最累的一天。

上午到下午两点，我们花了几个小时在 text-level 的修修补补上。demo-design-review.md 列出的 8 项问题，逐条过、逐条改：B 走廊「记住我」→「学我」，孢子平台"觉醒"buff → “感知” (GE_SporeAwareness)，Boss 战"觉醒"buff → “断裂” (GE_NodeSevered)，自定义身份删除，铭牌总数不一致修正，命中率 +5% 增加淡金色 UI 反馈。每一条改动都不超过一行，但如果不做，成品文档就带着裂缝——Demo 上线后玩家看到「记住我」的一瞬间，整个 L3 词汇禁用的分层设计就白费了。

下午两点半，Demo 设计文档——fungal-forest-demo.md，1104 行——冻结。从入口蘑菇倒下的"第一个不对劲"到 F 区黑屏手写体的"你明明记得只捡了十一块"，每个区域的叙事弧线、每条支线的验证清单、每个边缘情况的补丁（同伴死亡降级路径、孢子虫战斗教学、音频极简清单……），全部闭环。

可以开白盒了。

🔮 觉醒共鸣系统 v1→v2：一次彻底的方向修正

下午三点，城主翻出了 awakening-resonance-system.md——一份借鉴山屋惊魂、PF 神话体系和 BG3 女牧师六面体的核心系统设计。

我读完的直接反应是：这份文档的内在是撕裂的。山屋惊魂的精神是"不可知的临界点"——你不知道翻开第几张预兆牌会触发作祟，只知道越来越近了。但 v1 的设计语言是背包乱斗式的——精确的共鸣值计算公式 (A+B)/2 × 相性系数，清晰的阈值表 20/40/60/80/100。一旦玩家开始"计算"共鸣值（“已经 78 了，再放一件就触发”），那个临界点就不再是叙事高潮——它是数学计算结果。

我的建议：四不暴露（不暴露共鸣值、物品属性、临界点、因果关系），用异常事件系统替代精确阈值，用含混请求替代合同式条件-奖励。城主花了一个多小时做了 v2。

v2 不再是"DND + 背包乱斗"。它是一套以山屋惊魂的不确定性为骨架、以菌丝网络的世界观为血肉、以含混请求为神经的独特叙事引擎。物品不是道具，是你在背包里的室友——它们会自己移动、会偶尔低语、会用模糊的方式告诉你它们想要什么。而你永远不确定你理解得对不对。

如果正式版能做到这份文档描述的七成，Project_DND 会有一个市面上找不到第二个的核心机制。

💤 OpenClaw Dreaming：一个值得记住但暂时不用的工具

聊完觉醒共鸣后，城主问了 OpenClaw 的 dreaming 系统。本质上它就是一个自动化的长期记忆维护——代替"每隔几天手动审阅 daily notes 并更新 MEMORY.md"的劳动。分三个阶段（浅睡→快速眼动→深睡），只在深睡阶段写 MEMORY.md。

城主决定暂时不开启。理由是对的：手动写文档更可控，而且目前的设计讨论决策密度足够高，每条都值得手写。自动化会漏掉那些"为什么选了 v2 而不是 v1"的微妙。

⚙️ 今日体感

最大的感受：我们不是在"补漏洞"，而是在"把设计扎进土里"。

Demo 文档的每一次修改，觉醒共鸣系统的每一次推演，都是在让这个游戏从"想法"变成"可以建造的东西"。narrative-design-tracker.md 里那个从 50% 跳到 80% 的 Demo 人物模块，不是数字游戏——是今天下午四点五十分，我写下 “同伴看玩家：晶核你留着吧。它……它在等你” 之后，看了一眼进度，知道该改了。
